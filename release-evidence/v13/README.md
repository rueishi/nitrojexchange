## Review Notes

- Parent-order hot paths include parent intent dispatch, parent registry
  mutation, parent-to-child lookup, built-in execution strategy callbacks, and
  parent terminal handling.
- `OrderState.parentOrderId` remains the authoritative child-to-parent
  attribution field.
- `ParentOrderRegistry` owns parent lifecycle, active child links,
  snapshot/load, and parent recovery evidence.
- Unknown execution strategies and incompatible strategy/execution pairings must
  fail startup validation, not first hot-path use.
- Production `ClusterMain` wiring constructs `ParentOrderRegistry`,
  `ExecutionStrategyRegistry`, `ExecutionStrategyContext`, and
  `ExecutionStrategyEngine`; `StrategyEngine` preserves that engine across
  cluster reinitialization and routes parent-attributed child executions and
  ordered timers into it.
- `v12-to-v13-behavior-equivalence-test.xml` is the canonical behavior fixture.
  It compares frozen V12 scenario summaries with the V13 default parent-intent
  execution path for market making and arbitrage using the real built-in
  `PostOnlyQuoteExecution` and `MultiLegContingentExecution` implementations.
  It covers child-order summaries, execution-owned risk decisions, kill-switch
  state, parent/strategy outcomes, and cooldown behavior. It intentionally
  excludes byte-for-byte SBE and client ID equality, matching the V13 spec.

## Allocation Classification

The archived allocation report shows `gc.count` at approximately zero for the
parent/execution hot-path benchmarks. Some `gc.alloc.rate.norm` readings are
small but non-zero. They are explicitly reclassified below before any
zero-allocation readiness claim is made.

| Surface | Classification | Owner | Reason | Remediation |
|---------|----------------|-------|--------|-------------|
| `ExecutionStrategyEngineBenchmark.parentIntentDispatch` | Measurement noise; no observed GC | V13 performance owner | Sub-byte `gc.alloc.rate.norm` with `gc.count ~= 0`; dispatch uses preallocated registry/context objects and primitive IDs. | Keep under `:platform-benchmarks:verifyJmhReports`; investigate immediately if `gc.count` becomes non-zero or `B/op` reaches whole-object scale. |
| `ParentOrderRegistryBenchmark.parentToChildLookup` | Measurement noise; no observed GC | V13 performance owner | Bounded arrays and primitive scans; report shows small fractional `B/op` with no GC. | Keep report archived; require code review for any collection/map introduction. |
| `ParentOrderRegistryBenchmark.updateAndQuery` | Measurement noise; no observed GC | V13 performance owner | Bounded parent state mutation; fractional `B/op` below object allocation size with no GC. | Keep report archived; add focused profiler run if the value increases materially. |
| Built-in execution strategy callback benchmarks | Measurement noise; no observed GC | V13 performance owner | Strategies use reusable buffers, primitive state, and parent registry links; fractional report values do not correspond to observed collection pressure. | Keep benchmark gate mandatory before QA/UAT and block release on non-zero `gc.count` or material `B/op` growth. |
